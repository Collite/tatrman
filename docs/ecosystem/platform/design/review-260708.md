## 3. Decision review — pushback & gaps

### 3.1 The platform-managed world: assumed everywhere, designed nowhere — **the biggest structural gap**

E-2-γ registers engines as entries in "the **platform-managed world**." C-6 assembles snapshot archives "from the *resolved* world." The hero's life 2 serves "the world" from the metadata server. But:

- **Where does the platform-managed world live?** If text is canonical and registration = world content, admin actions produce edits to *some* world document in *some* repo. Which repo? Owned/reviewed by whom?
- **How does a project's world compose with it?** TTR-P T4 makes the world a *project* document. A project compiling connected must see platform-registered engines through the seam — so either the archive assembler *merges* platform-world entries into the project's resolved world (a merge nobody has designed: precedence, conflicts, `extends` vs `hosts:` mechanics), or projects `import`/extend the platform world explicitly (a language-adjacent surface TTR-P hasn't got).
- **Multi-project is the ordinary v1 case, not the parked multi-tenant case.** Two teams, two repos, one platform: do both declare the same Postgres instance in their own worlds (N truths — precisely the "world says PG-15, registry says PG-16" P3 catastrophe E-2-α was rejected for, reappearing *between projects*), or is the platform world the single source of instance truth that project worlds reference? The second is clearly the intent — but it's an inference, not a decision.
- Relatedly: `ttr-designer-server` is repo-attached, single-repo; the platform metadata server must serve **many** project repos plus the platform world. Its multi-repo model (registration of projects? discovery?) is unstated.

**Recommendation:** open this as a named thread — world composition (project world × platform world) — before or with G/I. It is C/E/G/I-joint and load-bearing for all of them. Most of the machinery likely exists (T4 `hosts:`, B-T6 layers, imports), but the composition rule must be a decision, not an accident.

### 3.2 F-2-δ "sound by construction" is overstated as recorded

The pointer-deploy variant (`{repo ref, commit, lock hash, program qname}`; platform compiles server-side) is called "sound by B-3-α." Not quite: **stats are not in the lock** (BQ-3). A pointer deploy that recompiles cites canon but *floats stats* — the platform's compile may pick a **different plan than the author reviewed locally**, which is exactly the reviewability failure B-2-δ was rejected for. δ is sound only if the pointer cites the full recorded snapshot: `{lock hash + compile record (recorded stats values)}` — i.e., the same provenance pair F-7 already established, with recompile-from-recorded-stats semantics. One sentence fixes it; record the correction before δ is ever implemented from the current text.

### 3.3 Plugin determinism: stated, not enforceable

E-1-β makes determinism "a stated SPI obligation" and BQ-4 pins plugin *identity* — but nothing verifies plugin *behavior*. In-tree targets are covered by `ttrp-conform`; a third-party/commercial plugin that timestamps its output silently breaks B-3-α parity for its users, and the WLA tier (the declared commercial-plugin market) is precisely where this will happen. Cheap mitigation exists because compiles are pure: a **double-compile byte-compare** (`ttr` verb or conform-shipped determinism kit) any plugin consumer can run, plus making it a plugin-certification requirement. Add to EQ-2's planning scope (it's trust-adjacent anyway).

### 3.4 F-4-iii resume vs staging lifecycle — an undesigned interaction

Wave-level resume ("completed islands skip iff snapshot unchanged") presumes the completed islands' *outputs* still exist when the run resumes. For islands landing in final targets, fine; for islands whose outputs are **staged** (Charon staging edges, F-b heritage), resume requires staging to *survive the failed run* — while retries (F-4-ii) require per-island staging *wipes*, and Charon has an Evict lifecycle nobody scoped platform-side. Staging retention/eviction across failed→resumed runs is a real semantics question (who owns staged data of a parked run? for how long?), not implementation detail. Name it as an F planning work item beside FQ-2/FQ-4.

### 3.5 No explicit v1 success criteria — the TTR-P A4 lesson recurring

TTR-P had A4 ("hero scenario in ≥2 surfaces, identical results") as the "done" bar; this effort has hero *lives* but no acceptance statement. Meanwhile v1 mass has quietly accumulated: new metadata server + program door + query door + transplants (C), trigger layer + executor + run store + quotas + envelope (F), **two full orchestrator support packages** plus the bash extraction (EQ-3), strangler ①–⑦ (D). Each increment was consciously ratified; the *sum* was never looked at. Before `/planning`: write the platform-v1 acceptance line (e.g., "life 2 end-to-end on a fresh deployment; life 3 via Airflow 3; Kestra proves the SPI" — or a deliberately weaker/stronger bar), so planning can sequence against a bar rather than a pile. This is the same shape as the previous review's §3.1 finding (scope larger than its own success criteria require) — it was right then.

### 3.6 H is now the pile-up — and secrets have no named mechanism

Riders routed to H: whois-descendant shape (C-5-ii) · ingress identity (C-5-iii) · scheduled-run principal (FQ-5) · advisory-only compile policy (B's last IOU) · registration secrets/connection-ref resolution (E-2-γ) · plugin signing/trust (EQ-2-adjacent) · quota governance (F-5-δ, parked) · Q-1 standalone security · LF-8 · RLS-in-v1 scope (map §H). That is a workstream-and-a-half. Two sharpenings: **(a)** connection-**ref** resolution is on the run path of everything (envelope bindings, Charon, harvest connectors) and no option for the secret store behind the refs has ever been catalogued — even the box deserves a name before planning; **(b)** E-3-β (native DAG, hall bypassed) is legal in standalone worlds — H should own the one-line governance story ("platform-governed engine credentials resolve platform-side only; standalone bypass = you already had the credentials"), so the bypass is bounded by design rather than by hope. **H should be the next divergence session.**
