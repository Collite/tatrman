# Tatrman Modeler v2.1 — Inline Mappings

**Status:** Design draft v1, 2026-05-27. Input to v2.1 phase planning. Grammar diff for ai-platform in [`grammar-v2-1-changes.md`](grammar-v2-1-changes.md). Implementation plan in [`../plan/implementation-plan-v2.1.md`](../plan/implementation-plan-v2.1.md).

## 1. Scope

v2.1 adds *syntactic sugar* for declaring entity-to-table mappings inline within `er` schema definitions, instead of forcing every mapping into a separate `def er2db_*` declaration in the `map` schema. The motivation is ergonomics: in real projects, the bulk of `def er2db_attribute` declarations are one-line, mechanically determined from an entity attribute and its target column. Forcing them into a separate file (and a separate schema) doubles the typing and splits the cognitive context.

This is **additive** — every existing `def er2db_*` declaration continues to work unchanged. Inline mappings synthesize the equivalent symbols at load time; the rest of the system (Designer, find-references, validators) sees no difference between an explicit and a synthesized mapping symbol.

The change spans the grammar, parser AST, semantic synthesizer, and conflict validator on the modeler side, and the equivalent layers on ai-platform's Kotlin side. It is a **minor version bump** (2.0 → 2.1) per the version scheme in `packages/grammar/CHANGELOG.md`: additive, no previously-valid `.ttr` file fails to parse.

## 2. Decisions captured

| #  | Decision                                | Choice                                                                                                       | Why                                                                                              |
| -- | --------------------------------------- | ------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------ |
| C1 | New keyword                             | `mapping`                                                                                                    | Reads naturally on `def entity X { ..., mapping: ... }`; not in current use.                     |
| C2 | Inner per-attribute map key             | `columns:` (not `attributes:`)                                                                               | The inner map's *values* are column references; reusing `attributes:` would shadow the entity's own attribute list. |
| C3 | Surface forms supported                 | Four: full nested, bare-id on attribute, bare-id-on-target shorthand, fully-wrapped attribute object         | See §3. Two clear extremes (full + bare id) plus one wrapper relaxation; no middle forms.       |
| C4 | Standalone `def mapping { type: ... }`  | **Deferred** to v3.0 (or beyond).                                                                            | Conceptually replaces `def er2db_*`; bigger change than syntactic sugar; deserves its own design.|
| C5 | Conflict between inline and explicit    | **Error** (`ttr/duplicate-mapping`).                                                                          | Silent override would hide an authoring bug. Explicit fix preserves authorial intent.            |
| C6 | Schema of synthesized `er2db_*` symbols | **Schemaless** — semantic-table entry only, not attributed to any source schema.                              | Inline mappings live in `.ttr` files declared as `schema er`; they don't belong to `map`, and forcing them there would imply an `import map` that the user didn't write. Synthesized symbols are not visible in `.ttrg` `objects:` lists and do not appear in the `map` schema's symbol table. |
| C7 | Source location of synthesized symbols  | Point back at the original inline `mapping:` block.                                                           | Cmd-click on `db.dbo.QZBOZI_DF` in the Designer's inspector must land in `er.ttr`, not at a synthetic position. |
| C8 | `whereFilter` in inline form            | **Not supported** in v2.1.                                                                                    | `whereFilter` is legacy; the user is removing it. No reason to plumb it through the inline form. |
| C9 | Relations supported                     | Yes — same four-form pattern as attributes.                                                                  | Parity. The inline relation form synthesizes `er2db_relation` with an `fk:` target.              |
| C10| `map` schema retained                   | Yes — explicit `def er2db_*` declarations remain valid indefinitely.                                          | Backward compatibility. Inline is sugar, not a replacement.                                       |

## 3. The four surface forms

For each kind (entity, attribute, relation) the grammar accepts exactly the shapes shown here. The semantic layer normalizes all of them into the equivalent explicit-form symbol table entries.

### 3.1 On an entity — full

```ttr
def entity artikl {
    description: "...",
    nameAttribute: název_artiklu,
    mapping: {
        target: { table: db.dbo.QZBOZI_DF },
        columns: {
            id_artiklu:     IDZBOZI,                           // form (a): bare id
            kód_artiklu:    { target: KOD_ZBOZI },             // form (b): wrapped, bare target
            název_artiklu:  { target: { column: NAZEV_ZBOZI } } // form (c): fully nested
        }
    },
    attributes: [
        def attribute id_artiklu     { type: int, isKey: true },
        def attribute kód_artiklu    { type: text },
        def attribute název_artiklu  { type: text }
    ]
}
```

Synthesizes:
- One `er2db_entity` for `artikl` → `db.dbo.QZBOZI_DF`
- One `er2db_attribute` per entry in `columns:`

Forms (a), (b), (c) are interchangeable per entry — the synthesizer treats them identically. The author chooses based on per-row complexity (e.g. (a) for trivial 1:1 mappings, (c) when a future per-column knob is anticipated).

### 3.2 On an attribute — bare id

```ttr
def attribute id_produktu {
    type: int,
    mapping: IDSKUPZBOZI
}
```

Synthesizes one `er2db_attribute` for `<enclosing-entity>.id_produktu` → the column reference, defaulting the kind to `column`.

### 3.3 On an attribute — full

```ttr
def attribute název_artiklu {
    type: text,
    mapping: { target: { column: NAZEV_ZBOZI } }
}
```

Equivalent to §3.2 with the wrapping made explicit. Useful when future per-attribute mapping properties land (none in v2.1).

### 3.4 On a relation — bare fk reference

```ttr
def relation artikl_produkt {
    from: er.entity.artikl, to: er.entity.produkt,
    cardinality: { from: "0..*", to: "0..1" },
    join: [{ from: er.entity.artikl.id_produktu, to: er.entity.produkt.id_produktu }],
    mapping: db.dbo.fk_artikl_produkt
}
```

Synthesizes one `er2db_relation` for `<relationName>` with `fk:` set to the FK reference.

### 3.5 On a relation — full

```ttr
def relation artikl_produkt {
    ...
    mapping: { fk: db.dbo.fk_artikl_produkt }
}
```

Equivalent to §3.4.

### Not supported (deliberately omitted)

- `mapping: { column: ... }` — target wrapper omitted. Two forms are enough; a third middle form makes "what does this mean" ambiguous, especially for relations.
- `id_artiklu: target { column: ... }` — bareword `target` followed by a block. Breaks the value-grammar shape; no other construct does this.
- `def mapping { type: er2db, ... }` — see C4.

## 4. Semantic model

### 4.1 Synthesis

At parse-resolve time, the semantic layer walks every `mapping:` property and produces equivalent symbol-table entries as if the user had written explicit `def er2db_*` declarations. The synthesized entries:

- Are added to the project's symbol table under the same qname they would have had if written explicitly: `<package>.map.er2dbEntity.<entityName>` for entities, `<package>.map.er2dbAttribute.<entityName>.<attrName>` for attributes, `<package>.map.er2dbRelation.<relationName>` for relations.
- Carry a `mappingSource: 'inline'` discriminator so the validator (§5) and the Designer can tell them apart from explicit declarations.
- Carry source locations pointing back at the inline `mapping:` block — specifically at the *value* of the `mapping:` property (the object or bare id), not the keyword. This is what enables Cmd-click navigation in the Designer to land on the actual authored text.

Per C6, the synthesized entries do **not** belong to the `map` schema's per-file symbol table. They are semantic-table only — invisible to `.ttrg` `objects:` lists and not subject to `map`-schema-only validators. The `map` schema continues to host explicit `def er2db_*` declarations as today.

### 4.2 Naming subtlety: synthesized qnames

Synthesized symbols use the *host file's package*, not the `map` schema's package. A file `billing/products/er.ttr` declaring `package billing.products` and containing `def entity artikl { mapping: ... }` synthesizes `billing.products.map.er2dbEntity.artikl`. This means cross-package entity collisions on the bare name (`artikl` in `billing.products` and `artikl` in `billing.invoicing`) do *not* collide at the mapping level — each lives under its own package prefix.

**Namespace assumption.** The synthesizer produces qnames using the AST `kind` token in camelCase (`er2dbEntity`, `er2dbAttribute`, `er2dbRelation`) — matching what `addEntry` produces for an explicit `def er2db_entity X` in a `schema map` file **with no namespace declared**. If a project's `map.ttr` declares a namespace (`schema map namespace <X>`), `addEntry` produces `<pkg>.map.<X>.<name>` and synthesized symbols land at a different qname; `duplicates()` and the `ttr/duplicate-mapping` validator will be blind to those collisions. The production convention (see `samples/v1.1-metadata/billing/map.ttr`) is no namespace on map-schema files, so this is acceptable for v2.1. Defer kind-stable er2db_* qnaming to a future cleanup.

### 4.3 Cross-schema dependency

The moment a file contains an inline mapping, it depends on the `db` schema at resolve time (because `db.dbo.QZBOZI_DF` is a reference into the `db` schema). Files using inline mappings must import the target schema. For files in the same package as their target db tables, no import is needed (same-package rule); otherwise an `import` is required, same as for any cross-package reference.

This is a real architectural shift acknowledged by the author: the `er` schema in projects that use inline mappings is no longer "pure conceptual model" — it has a hard dependency on the physical model. Projects that want a `er`-only deliverable should continue to use explicit `def er2db_*` in a separate `map.ttr`.

## 5. Conflict semantics

A *duplicate mapping* is any of:

1. **Inline + explicit collision.** Inline `mapping:` on `def entity X` *and* explicit `def er2db_entity X` for the same entity, in any file.
2. **Two-inline collision** (theoretically possible only across files where the same entity is referenced — not applicable in v2.1 because entities are package-local). Reserved for future use.
3. **Inline-attribute + explicit-attribute collision.** Inline `mapping:` on `def attribute Y` *and* explicit `def er2db_attribute X.Y` for the same enclosing entity.
4. **Inline-entity-with-columns + explicit-attribute collision.** Inline entity-level `mapping.columns.Y` *and* explicit `def er2db_attribute X.Y`.
5. **Inline-relation + explicit-relation collision.** Same pattern for relations and their FKs.

All produce a single new diagnostic:

| Code                       | Severity | When                                                                                  |
| -------------------------- | -------- | ------------------------------------------------------------------------------------- |
| `ttr/duplicate-mapping`    | Error    | Inline and explicit mappings both target the same er2db_entity / er2db_attribute / er2db_relation |

The diagnostic fires on **both** locations (both `mapping:` value and the explicit `def er2db_*` header), so the user sees the conflict from either side. The message includes the qname and the two source locations.

## 6. Backward compatibility

- Every existing v2.0 `.ttr` file parses unchanged under v2.1. The `mapping:` property is new and optional; no existing file uses it.
- Every existing `def er2db_*` declaration continues to work. The `map` schema is not deprecated.
- The TextMate grammar gains `mapping` and `columns` (when used as the inner key) as keyword scopes; existing files with no inline mappings see no highlighting changes.

The grammar version bumps `2.0 → 2.1` (minor — additive, no breakage).

## 7. ai-platform coordination

Per the same pattern as v1.1, ai-platform's Kotlin parser regenerates from the vendored grammar and the loader's metadata path picks up the new shapes. Because v2.1 is additive, ai-platform's adoption is not on the critical path for modeler v2.1's release — but unlike v1.1, this PR aims to **ship both sides together** (single deployment unit per the user's direction). The coordination doc is [`grammar-v2-1-changes.md`](grammar-v2-1-changes.md).

The ai-platform-side work mirrors modeler's Sections C–E (parser AST, semantic synthesizer, conflict validator). Tests on the Kotlin side cover the same shapes as the modeler-side integration tests.

## 9. Open questions

These are flagged for resolution during implementation; none block Section A from landing.

1. **`columns` as a context-sensitive keyword.** `columns` is already in use as a property of `tableProperty` (the list of column defs in a table). Using `columns:` as a key inside the inline-mapping object is grammatically fine because it appears in a generic `object_` context, but it does mean `columns` is now meaningful in two places. Decide during Section B whether to add `COLUMNS` to `idPart` (probably already there via the `COLUMNS` token).
2. **Synthesized qname source-location for "find references".** When the user runs find-references on `db.dbo.QZBOZI_DF`, do the references include synthesized inline mappings? Yes per C7. But the LSP's reference list shows the synthesized er2db_entity's location, which points at the inline `mapping:` value — that's the right place. Worth double-checking in Section D.
3. **Designer "inspector — referenced by" list.** When the inspector shows what references a db table, do inline mappings appear distinguishably from explicit ones? Probably useful — a small icon or label. UI polish, defer to v2.1.1 or v2.2.
