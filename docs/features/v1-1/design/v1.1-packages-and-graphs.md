# Tatrman Modeler v1.1 — Packages, Imports, and Graph Files

**Status:** Design draft v1, 2026-05-18. Input to v1.1 phase planning; supersedes parts of `docs/v1/design/architecture.md` (notably §6 on `.ttrl`). Revise as items in §13 close.

## 1. Scope

v1.1 introduces three interlocking changes that, together, give TTR a real modularity story and give the Modeler the ability to render *named slices* of a project rather than "everything in the folder":

1. **Packages.** Every `.ttr` file declares the package it belongs to; the directory layout mirrors the package hierarchy.
2. **Imports.** Cross-package references require an explicit `import`. Same-package references do not. This makes the dependency graph between packages explicit and inspectable.
3. **`.ttrg` graph files.** A new file type — the *Tatrman Graph* — that declares an explicit, version-controlled slice of the model: its scope (which packages it draws from), its members (which objects it shows), and its layout. `.ttrg` replaces `.ttrl` entirely.

This is a deep language change, not just editor tooling. The grammar gets new keywords, the resolver changes its rules, the symbol-table qname shape changes, and the Designer's "load a folder" workflow is replaced with "load a graph file or create a new one." Because the change is at the language level, ai-platform's Kotlin parser must adopt it in lockstep (decision B1).

This document is the design specification. The phased work plan lives separately in [`docs/v1-1/plan/implementation-plan-v1.1.md`](../plan/implementation-plan-v1.1.md).

## 3. Package model

### 3.1 Declaration

Every `.ttr` and `.ttrg` file begins with a `package` declaration:

```
package billing.invoicing
schema er namespace entity

def entity artikl { ... }
```

The declaration appears before `schema`. (Open: B5 mandates the declaration; the placement *relative to* `schema` could equally well be before or after. v1 syntax has `schema X namespace Y` as the first line; v1.1 prepends `package X.Y.Z`. See §13.)

### 3.2 Package = directory

> **Revised by [§14](#14-addendum-2026-06-19--nested-packages-finalised-root-prefix-no-cascade-domains) (B15–B18).** The in-file `package` declaration is now *authoritative*; the directory is a *checked convention*. A mismatch is no longer an unconditional error — its severity is the configurable `[packages].layout` knob (default Warning), and a configurable `root` prefix is prepended to directory-derived names. Read §14.1–§14.3 alongside this section.

The directory containing `modeler.toml` is the single classpath root. A file's package is determined by its path relative to that root:

```
<project-root>/
  modeler.toml                                    # root marker
  billing/
    invoicing/
      er.ttr             → package billing.invoicing
      db.ttr             → package billing.invoicing
      graphs/
        artikl_view.ttrg → package billing.invoicing.graphs
    products/
      er.ttr             → package billing.products
```

If the declared package doesn't match the directory, the file produces a `ttr/package-declaration-mismatch` diagnostic.

### 3.3 The default (empty) package

Files at the project root with no subdirectory have no declared package — `package` keyword is omitted, and they are in the *default package*. This is supported but discouraged (matching Java); a `ttr/missing-package-declaration` info-level diagnostic flags it. Migration tooling won't *create* a default-package file; it's only there to keep small/exploratory projects friction-free.

### 3.4 Naming convention

Lowercase, dot-separated, identifier rules per the existing TTR `IDENT` token (Latin Extended + underscore + digits, not starting with a digit). We *recommend* but do not enforce reverse-DNS (`com.tatrman.billing.invoicing`); short names like `billing.invoicing` are valid.

### 3.5 Duplicate-definition resolution

Two defs with the same simple name in the same package collide and produce `ttr/duplicate-definition` (already shipped in v1). The same simple name in *different* packages does **not** collide — they are distinct symbols (`billing.invoicing.er.entity.Invoice` ≠ `accounting.er.entity.Invoice`).

## 4. Import model

### 4.1 Syntax

Imports come immediately after the package declaration, before the `schema` directive:

```
package billing.invoicing

import billing.products.*
import accounting.er.entity.Invoice

schema er namespace entity

def entity artikl { ... }
```

Two forms:

| Form                                       | Meaning                                                          |
| ------------------------------------------ | ---------------------------------------------------------------- |
| `import <package>.*`                       | Wildcard: all top-level defs in `<package>` become bare-resolvable |
| `import <package>.<schema>.<namespace>.<defName>` | Named: this specific qname becomes bare-resolvable               |

(The named form is the package-prefixed qname per B7. We don't allow `import <package>.<defName>` as a shorthand — the schema and namespace must appear.)

Aliases (`import X as Y`) are deferred. See §13.

### 4.2 Resolution rules

The resolver tries, in order:

1. **Lexical scope** — attribute-within-entity, column-within-table, role-within-stock. (Unchanged from v1.)
2. **Same package** — any def in the same package is resolvable by its bare name without an import. This is the "you don't import your siblings" rule from Java.
3. **Named imports** — exact-match qname imports.
4. **Wildcard imports** — top-level defs in any wildcard-imported package.
5. **Auto-imports** — the `cnc.*` stock vocabulary (B8) is always in scope.
6. **Fully qualified name** — `<package>.<schema>.<namespace>.<defName>` always resolves regardless of imports, provided the symbol exists.

Failure produces `ttr/unresolved-reference` (existing) with a new `tried[]` entry shape that includes which packages were searched and why each was rejected ("not imported", "wildcard non-recursive", etc.).

### 4.3 Wildcard non-recursion

`import billing.*` pulls in defs from package `billing` only — not from `billing.invoicing` or `billing.products`. To get those, import them explicitly. (B9, matches Java.)

### 4.4 Auto-imported stock vocabulary

The `cnc` package (the six built-in CNC roles: `fact`, `dimension`, `structural`, `master`, `transaction`, `bridge`) is implicitly imported in every file as if `import cnc.*` were the first import. (B8.)

If a project ever introduces a non-cnc stock vocabulary (deferred; not in v1.1), the auto-import set will be configurable via `modeler.toml` `[stock].auto-import`.

### 4.5 New diagnostics around imports

| Code                                  | Severity | Trigger                                                                                  |
| ------------------------------------- | -------- | ---------------------------------------------------------------------------------------- |
| `ttr/unimported-reference`            | Error    | Bare reference to a def that exists but is in a non-imported package                     |
| `ttr/unused-import`                   | Warning  | An `import` statement whose targets are never referenced                                 |
| `ttr/wildcard-with-no-matches`        | Warning  | `import x.y.*` where package `x.y` has no defs (typically a typo'd package name)         |
| `ttr/duplicate-import`                | Warning  | The same package is imported twice (wildcard or named), or named import shadows wildcard |
| `ttr/circular-package-dependency`     | Warning  | Package A imports B, B imports A — not an error (it's legal) but worth flagging          |
| `ttr/package-declaration-mismatch`    | Error    | Declared package doesn't match the file's directory                                      |
| `ttr/missing-package-declaration`     | Info     | File is in the default package (no `package` keyword)                                    |
| `ttr/ambiguous-reference`             | Error    | Bare reference matches defs in two or more wildcard-imported packages                    |

All codes follow the existing `ttr/*` taxonomy (`docs/v1/design/diagnostics.md`).

## 5. Qname shape

v1 qname: `<schema>.<namespace-or-kind>.<defName>[.<subDef>]` — e.g., `er.entity.artikl`, `db.dbo.QZBOZI_DF.IDZBOZI`, `cnc.role.fact`. The second component is the schema's declared `namespace` if present, otherwise the def kind (which is why `cnc` files with no namespace yield `cnc.role.*`, `cnc.fact.*` etc.).

v1.1 qname: `<package>.<schema>.<namespace-or-kind>.<defName>[.<subDef>]` — e.g., `billing.invoicing.er.entity.artikl`, `billing.invoicing.db.dbo.QZBOZI_DF.IDZBOZI`. The auto-imported stock vocabulary lives in package `cnc`, so `cnc.role.fact` becomes `cnc.cnc.role.fact` — see open question §13.10 about whether to special-case this.

This is the canonical form everywhere the symbol table, LSP responses, `.ttrg` `objects` list, and hover/find-refs/go-to-def output appear. Bare references (in user-authored TTR text) continue to use the *shortest unambiguous form* per the resolution rules in §4.2.

**Migration implication.** Every existing cross-reference in user projects keeps working *after* migration tooling injects the right imports — they don't have to be rewritten to the new long form. The qname change is internal to the LSP/symbol-table; user-facing text stays as short as the resolution rules allow.

## 6. `.ttrg` — the Tatrman Graph file

### 6.1 Structure

One graph per file. The file's name (sans extension) is the graph's name. The file has the same package/import preamble as `.ttr`, followed by a single `graph` block:

```
package billing.invoicing.graphs

import billing.invoicing.*
import billing.products.*

graph artikl_overview {
    schema: er,
    description: "Overview of artikl entity and its direct relations",
    tags: ["billing", "core-domain"],

    objects: [
        billing.invoicing.er.entity.artikl,
        billing.products.er.entity.produkt,
        billing.products.er.entity.podprodukt,
        billing.invoicing.er.relation.artikl_produkt,
        billing.invoicing.er.relation.artikl_podprodukt
    ],

    layout: {
        viewport: { zoom: 1.0, panX: 0, panY: 0, displayMode: "with-types" },
        nodes: {
            "billing.invoicing.er.entity.artikl":       { x: 320, y: 180 },
            "billing.products.er.entity.produkt":       { x: 580, y: 180 },
            "billing.products.er.entity.podprodukt":    { x: 580, y: 380 }
        },
        edges: {}
    }
}
```

### 6.2 Properties

| Property      | Required? | Type                | Notes                                                                                          |
| ------------- | --------- | ------------------- | ---------------------------------------------------------------------------------------------- |
| `schema`      | Yes       | Schema kind (`er` / `db` / future)  | Locks the graph to one schema kind (B3)                                                  |
| `description` | No        | Localized string    | Same shape as on `def entity`/etc.                                                              |
| `tags`        | No        | List of strings     | For filtering in the Designer's graph picker                                                   |
| `objects`     | Yes       | List of qnames      | Each must resolve in the project; unresolved entries produce `ttr/graph-object-not-found` warning |
| `layout`      | No        | Layout object       | Same shape as the old `.ttrl` per-viewport block; missing nodes get auto-layout on open. **Superseded by the `.ttrl` sidecar per §15 (2026-07-03); remains the shipped mechanism until migration.** |

### 6.3 Object resolution and staleness

Every qname in `objects` is a cross-reference resolved by the standard resolver (§4.2). If the user renames `er.entity.artikl` → `er.entity.artikl_v2`, the `.ttrg` reference goes stale and produces `ttr/graph-object-not-found` (Warning).

The Designer surfaces this in a "missing objects" badge on the canvas. The user can choose to **remove** the stale entry (edits the `.ttrg`), or use `textDocument/rename` (v1.1 productivity tier) which propagates the rename into the `.ttrg`'s `objects` list automatically. This is the second concrete payoff from B10 — the `.ttrg` participates in standard LSP rename for free.

### 6.4 Edge inclusion semantics

For schema `er`: an edge (relation) appears in the graph if and only if **both endpoints are in `objects`** AND the relation itself is in `objects`. Same for `db` (foreign keys): both endpoint tables and the fk itself must be listed.

This makes graphs explicit rather than computed-by-traversal — what you see is exactly what's listed. (Alternative considered: auto-include all relations between selected entities. Rejected because it makes graph membership ambiguous and breaks "the file is the truth.")

### 6.5 Layout

> **Amended 2026-07-03 — see [§15](#15-addendum-2026-07-03--family-wide-ttrl-view-state-sidecar-amends-b2-62-layout-65).** Layout migrates from the in-file block below to a per-document `.ttrl` sidecar (family-wide with TTR-P). The block below remains the shipped v1.1 mechanism until that migration lands.

The `layout` block is the per-graph evolution of v1's `.ttrl`:

- `viewport.{zoom, panX, panY, displayMode}` — same shape as before
- `nodes` — keyed by qname, `{x, y}` per node
- `edges` — keyed by qname, `{bendPoints: [[x, y], ...]}` per edge (optional bend metadata)

Unlike `.ttrl`, layout is per-graph rather than per-project. There's no project-wide layout file. The Designer manages this via existing `modeler/setLayout` / `modeler/getLayout` machinery; signatures gain a `graphUri` parameter.

## 7. Grammar additions

Sketch of the additions to `TTR.g4` (final spelling settled during Phase 1.1.A):

```
document
    : packageDecl? importDecl* (schemaDirective | graphBlock)? definition* EOF
    ;

packageDecl
    : 'package' qualifiedName
    ;

importDecl
    : 'import' qualifiedName ('.' '*')?
    ;

graphBlock
    : 'graph' IDENT '{' graphProperty (',' graphProperty)* '}'
    ;

graphProperty
    : IDENT ':' propertyValue
    ;

qualifiedName
    : IDENT ('.' IDENT)*
    ;
```

New reserved keywords: `package`, `import`, `graph`. (Today, all three are likely usable as identifiers in `def`-name positions; v1.1 forbids that.)

`.ttrg` uses the same parser as `.ttr` — there is no separate grammar. The host distinguishes by file extension and the parser dispatches on whether the document contains a `graphBlock` vs. `definition*`. (A `.ttrg` file with no `graph` block, or a `.ttr` file with one, produces a `ttr/wrong-file-kind` error.)

## 8. Impact on existing v1 components

### 8.1 `@modeler/parser`

- AST: new `PackageDecl`, `ImportDecl`, `GraphBlock` nodes; `Document` gains optional `packageDecl`, `imports[]`, `graph?`
- CST view: trivia around package/import statements treated the same as around defs

### 8.2 `@modeler/semantics`

- Symbol table: qnames now package-prefixed (per §5); `DocumentSymbolTable` gains a `packageName` field; `ProjectSymbolTable` keys symbols by full qname
- Resolver: new resolution chain per §4.2; `ResolutionResult.tried[]` entries gain `reason` field
- Validator: new diagnostic codes per §4.5 and §6.2
- New module: `PackageGraph` — builds the directed dependency graph between packages from imports; used for circular-dep detection and (later) for the LSP's `modeler/getPackageGraph` method
- Stock vocabulary: now in package `cnc`, auto-imported via §4.4

### 8.3 `@modeler/lsp`

New custom methods:

| Method                                   | Purpose                                                                                            |
| ---------------------------------------- | -------------------------------------------------------------------------------------------------- |
| `modeler/listGraphs`                     | Returns all `.ttrg` files in the project with metadata (name, schema, description, object count)   |
| `modeler/getGraph(uri)`                  | Returns the resolved graph: schema, objects (with their per-kind data), edges, layout              |
| `modeler/addObjectToGraph(uri, qname)`   | Synthesizes a `WorkspaceEdit` that adds `qname` to the graph's `objects` list (+ import if needed) |
| `modeler/removeObjectFromGraph(uri, qname)` | Synthesizes a `WorkspaceEdit` that removes `qname` from `objects` (and the import if now unused) |
| `modeler/createGraph(packages, objects, schema, fileUri)` | Generates a new `.ttrg` file with the given scope and members                        |
| `modeler/getPackageGraph()`              | Returns the directed graph of package dependencies (for visualization in graph-creation UX)        |

Changes to existing methods:

- `modeler/getLayout` / `modeler/setLayout` — gain a `graphUri` parameter; the project-wide layout case goes away
- `modeler/getProjectInfo` — gains `packages: PackageInfo[]` field listing every package in the project with file counts and dependents
- `modeler/getModelGraph` — semantics shift from "render the whole project as a graph" to "render this `.ttrg` as a graph"; takes a `graphUri` parameter

### 8.4 `@modeler/vscode-ext`

- Language registration for `.ttrg` (separate language id `ttrg`, sharing TextMate scopes with `ttr`)
- File icon for `.ttrg`
- Drop `.ttrl` language registration entirely

### 8.5 `@modeler/designer`

The Designer's entry point changes. Today: open a folder → render its db/er. v1.1: three entry modes.

| Entry mode               | UX                                                                                                 |
| ------------------------ | -------------------------------------------------------------------------------------------------- |
| **Open existing graph**  | File picker → select a `.ttrg` → Designer renders it                                                |
| **Browse project graphs**| Open folder → `modeler/listGraphs` shows a graph picker → user selects one → Designer renders it    |
| **Create new graph**     | Multi-step wizard: (1) pick packages, (2) shown dependency tree, optionally add transitive packages, (3) pick objects from selected packages, (4) pick schema, (5) name + save as `.ttrg` |

In-canvas affordances added:

- **Add object** — opens a picker scoped to imported packages; selecting an object dispatches `modeler/addObjectToGraph`
- **Remove object** — context-menu item on any node; dispatches `modeler/removeObjectFromGraph`
- **Extend imports** — surfaced when the user picks an object outside the current import scope; offers to add the import automatically

The schema/displayMode toggle UI from Phase 3 stays, but schema toggling now means "switch which `.ttrg` is open" (since one graph = one schema per B3). The Designer remembers per-graph displayMode.

Layout persistence rewires from `.ttrl` to per-`.ttrg` layout blocks. The existing debounce/save logic is unchanged in shape — just the wire payload changes.

### 8.6 `modeler.toml`

No new required fields. The `[schemas].declared` array becomes informational only (the actual schemas in use are derived from package contents). The `[language]` block is unchanged.

Optional new field for projects that want to constrain auto-imports:

```toml
[stock]
auto-import = ["cnc.*"]   # default; can be overridden once non-cnc stock exists
```

## 9. Migration tooling

`packages/migrate/` — a new package — ships a CLI: `modeler migrate-to-packages <project-root>`.

The tool:

1. **Walks** every `.ttr` file under the project root
2. **Infers** the package name from the file's directory relative to `modeler.toml`'s location
3. **Inserts** `package <name>` at the top of every file (idempotent — skips files that already declare a package)
4. **Scans** every cross-reference; for each one whose target lives in a different package, inserts the appropriate `import` statement
5. **Converts** any existing `.modeler/layout.ttrl` into one or more `.ttrg` files. By default it produces two: `<project-root>/graphs/_all_db.ttrg` and `<project-root>/graphs/_all_er.ttrg`, each listing every object of that schema and carrying the layout positions from the old `.ttrl`. The user can then split or rename these graphs interactively in the Designer.
6. **Reports** what it did: files touched, packages created, imports inserted, ambiguous references that need manual disambiguation

The tool is idempotent and produces a `--dry-run` report mode. It does not delete the `.ttrl` until the user passes `--commit-ttrl-removal`.

ai-platform will run this tool on its own projects on their own schedule (B11). The modeler repo's `samples/` get migrated as part of v1.1 itself.

## 10. ai-platform coordination

Per B1, the grammar change must propagate to ai-platform's Kotlin parser. The coordination shape:

1. Modeler lands the grammar changes on a feature branch; CI verifies the TS parser regenerates cleanly
2. Modeler bumps the grammar version (e.g., `2.0.0` — major bump because of the new keywords)
3. Bora coordinates ai-platform's grammar bump; ai-platform's Kotlin parser regenerates and adopts the new resolution rules
4. Modeler's grammar-sync CI continues to compare hashes; ai-platform's vendored copy claims grammar `2.0.0` once it's ready
5. **v1.1 of modeler ships independently** of ai-platform's adoption schedule (B12). The grammar-sync CI failing because ai-platform is behind is treated as informational, not blocking, for the duration of the coordination window.

Coordination doc that accompanies this design: [`grammar-v1-1-changes.md`](grammar-v1-1-changes.md) — a precise diff of `TTR.g4` between v1 and v2.0 plus the resolution-rule changes, written for the ai-platform parser maintainer.

## 12. Relationship to v1.2 (graphical editor)

v1.1's `.ttrg` model directly enables v1.2's editor. Specifically:

- "Edit this graph" has a precise meaning: edit the objects listed in the `.ttrg`, plus their direct properties
- The edit synthesizer (placeholder in v1, partially exercised by v1.1's rename + add-object-to-graph) is fully load-bearing by v1.2
- Designer edit operations from v1.2 (add attribute, create relation, etc.) operate on the `.ttrg`'s scope — there's no ambiguity about "where" the new attribute lives

v1.1 also delivers the `WorkspaceEdit` synthesizer for the rename + graph-mutation paths, which means v1.2 inherits a working, tested synthesizer rather than starting from a `{ ok: false, ... }` placeholder.

## 13. Open questions (to revisit during implementation)

1. **Placement of `package`/`import` relative to `schema`.** Current draft: `package` → `import`s → `schema` → defs. Alternative: `package` → `schema` → `import`s → defs. The first is more Java-like; the second keeps the v1 line `schema X namespace Y` closer to the top. Decide during 1.1.A with a quick poll of which reads more naturally to ai-platform's TTR authors.
2. **Aliases (`import X as Y`).** Deferred. Revisit if v1.1's review surfaces name-collision pain.
3. **Multiple classpath roots.** v1.1 assumes one classpath root (the `modeler.toml` directory). Multi-root would let a project layer `src/` and `vendor/` etc. Deferred to v2.x unless an early user actually needs it.
4. **`.ttrg` `import` vs `objects` qname forms.** The current draft requires fully-qualified names in `objects` for clarity. We could relax to allow bare names when the import scope is unambiguous, matching `.ttr` rules. Decide during 1.1.C — there's a readability vs. consistency trade-off either way.
5. **`graph` as a sub-block of `.ttr`.** Should we allow a `graph` block inside a `.ttr` file (for tiny projects that don't want a separate `.ttrg`)? Current draft says no (B13). Revisit if migration of trivial projects feels heavyweight.
6. **Edge inclusion: explicit vs. computed.** §6.4 specifies explicit (relation must be listed). Alternative: auto-include relations whose endpoints are both in `objects`. The explicit choice is in keeping with "the file is the truth"; if it turns out to be annoying in practice, we can add an `auto-edges: true` graph property.
7. **Stale-qname garbage collection in `.ttrg`.** When the resolver marks a `.ttrg` object as missing, should the Designer offer one-click removal, defer to manual fix, or do both? Probably both, with the auto-fix gated behind a confirmation. Settle during 1.1.E.
8. **What happens to circular package dependencies at the resolver level.** Currently flagged as a warning (`ttr/circular-package-dependency`). Should they be errors? Java tolerates them (with caveats). I'd keep them as warnings for v1.1 — TTR's references are pure value references with no initialization order to worry about.
9. **`modeler.toml` `[stock].auto-import` shape.** Sketched in §8.6 but not strictly needed in v1.1 since only `cnc.*` is auto-imported. Whether to ship the config knob now or defer until a second stock vocabulary exists.
10. **Stock-vocabulary qname under the new package shape.** The v1 stock-cnc file is `schema cnc` with no namespace clause, producing qnames like `cnc.role.fact`. Under v1.1 it would naturally live in package `cnc` → `cnc.cnc.role.fact`, which is ugly. Three options: (a) accept the doubled `cnc`, (b) special-case: when the file's package name equals the schema name, drop the duplication, or (c) move stock files into a different package convention (e.g., empty/root package, or `stock`). Option (b) feels cleanest but adds a wrinkle to the qname rule. Decide during 1.1.B. **Update (B23, §14):** live `model-ttr` data already writes the doubled option-(a) form (`role: cnc.cnc.role.structural`), and contracts §3.1 mandates accepting it for v1.1. Keep option (a) for now; option (b) becomes a separately-tracked cleanup with its own migration, not part of the Packages & Domains increment.

These don't block 1.1.A; they get resolved as the relevant sub-phase lands.

## 14. Addendum (2026-06-19) — nested packages finalised, root prefix, no-cascade, domains

This addendum captures a second brainstorm that finalised the package model for real-world use, after `ai-platform`'s sibling repo [`ai-models`](https://github.com/Tatrman/ai-models) became a live consumer of the package concept. It is **input to a dedicated implementation increment** ("Packages & Domains", phases PD1–PD5) planned under [`docs/v1-1/plan/packages-domains/`](../plan/packages-domains/README.md). Decisions extend the B-series; where they touch §3–§5 the affected clauses carry inline pointers.

### 14.0 Why this addendum exists — the `ai-models` reality check

`ai-models/model-ttr/` is already a v1.1-shaped project (one directory per package, each holding `er.ttr` / `db.ttr` / `cnc.ttr`, every file declaring `package <name>`). Two things there forced these decisions:

1. **`ai-models` needs nested packages.** Confirmed by Bora 2026-06-19. The flat single-segment layout (`ucetnictvi`, `artikl`, …) will grow sub-domains.
2. **`ai-models` already consumes "package" in two more places** that the original design didn't account for:
   - the **agent registry** (`agents/*.yaml`), where `shem.packages: [...]` and `shem.entities: [pkg.entity]` scope what each Golem agent loads;
   - the **CI gate** (`tools/validate_agents.py`), which today equates "package" with "a top-level directory under `model-ttr/`" and parses entity refs with `split(".", 1)`. Both assumptions are nesting-hostile and must change. See §14.5.

### 14.1 Derivation algorithm (B15, B17, B18)

For a `.ttr`/`.ttrg`/`.ttrd` file at path `P` under project root `R` (the directory holding `modeler.toml`), with configured root prefix `ROOT` (empty if unset):

```
derivedPackage(P) = join(".", filter-nonempty([ROOT, relDirSegments(R, P)]))
```

where `relDirSegments(R, P)` is the directory path from `R` to `P`'s parent, split on `/`. The file's **effective package** is then:

1. If the file declares `package X` → effective package is `X` (**declaration authoritative**, B15).
2. Else → effective package is `derivedPackage(P)` (B15 fallback).

`derivedPackage(P)` is also what the *mismatch check* compares a present declaration against. Severity of a mismatch is the `[packages].layout` knob (B16).

**No cascade (B18).** When computing the effective package of a *child* directory, the algorithm never consults a parent file's declaration — it always uses `ROOT + path`. Consequence: a declaration that overrides only its **leaf** segment is harmless (the subtree below it is empty of derivation dependencies), but a declaration whose **non-leaf (prefix) segments** diverge from its path orphans the file from anything resolving through that path. That specific case earns a louder diagnostic, `ttr/package-prefix-divergence` (Warning even in `flexible`; Error in `strict`), separate from the ordinary leaf-only `ttr/package-declaration-mismatch`.

**Segment validity (B24).** Package names are dot-separated `IDENT`s; `IDENT` is `[a-zA-ZÀ-ɏ_][a-zA-Z0-9_À-ɏ]*` — letters, digits, underscore, **no hyphen** (same as Java/Go segments). Every segment produced by `derivedPackage(P)` must be a valid `IDENT`. There is **no** `-`→`_` (or any other) normalization: silent rewriting would collapse `my-pkg/` and `my_pkg/` into one package and hide the divergence, so we reject instead of rewrite. Rules:

- If a path segment is not a valid `IDENT` (hyphen, space, leading digit, …) **and** the file has no `package` declaration → `ttr/invalid-package-segment` (Error under `strict`, else Warning). The author must rename the folder (preferred — the project convention is underscores, e.g. `obchodni_doklady`) or add an explicit declaration.
- If such a file **does** declare a valid `package` → the declaration wins (B15) and the ordinary `ttr/package-declaration-mismatch` / `ttr/package-prefix-divergence` checks are **suppressed** for the offending segment, because the directory was never a valid candidate to compare against. Only `ttr/invalid-package-segment` would have fired, and the declaration resolves it. This is the sanctioned way to keep a human-friendly hyphenated folder name (`my-cool-pkg/`) behind a valid package (`package my_cool_pkg`).

**Guidance to authors:** keep folders == packages, **use underscores** in folder names (project convention; matches `model-ttr`); treat the declaration override as an escape hatch, not a structuring tool. To rename a whole subtree, rename the folder or change `ROOT`.

### 14.2 `modeler.toml` additions (B16, B17)

```toml
[packages]
# Optional module-style prefix prepended to every directory-derived package name.
# Elidable in references: `package ucetnictvi` and `ucetnictvi.er.entity.faktura`
# stay legal even though the canonical qname is `com.tatrman.ucetnictvi.…`.
root = "com.tatrman"        # default: "" (no prefix)

# How strictly a present `package` declaration must match its directory:
#   "flexible" — mismatch is a Warning (default); declaration wins
#   "strict"   — mismatch is an Error
#   "off"      — no mismatch diagnostic at all (declaration always wins)
layout = "flexible"          # default: "flexible"
```

`root` elision rule: a reference (or an agent's `packages:`/`entities:` entry, or a `.ttrd` member) that omits the `root` prefix resolves as if the prefix were present. The canonical (symbol-table) qname always carries the prefix; user-authored text may omit it. This is what lets `ai-models` keep writing bare `ucetnictvi` while the canonical name gains a prefix if one is ever configured.

### 14.3 Reference recursion: load vs import (B20)

Two prefix-matching semantics coexist and must not be conflated:

- **`import X.*`** (in `.ttr`/`.ttrg`/`.ttrd`) is **non-recursive** (B9, unchanged): exposes top-level defs of package `X` only, not `X.sub`.
- **domain membership "load `X`"** (in `.ttrd`) is **recursive**: pulls package `X` and every `X.*` descendant.

These are different operations with different intents (compile-time reference resolution vs runtime scope selection). The recursion lives *only* in domain membership, so nowhere does a single `X.*` token mean two things.

### 14.4 Domains — the `.ttrd` file (B19, B20, B21)

A **domain** is a named, curated, recursive grouping of model packages and individual entities, used to scope what a downstream consumer (a Golem agent) is allowed to see. Domains are orthogonal to packages: a package is the unit of import resolution; a domain is the unit of consumer scoping. A domain name never appears in a qname.

**File kind.** Domains live in `.ttrd` files, parsed by the same engine as `.ttr`/`.ttrg` (consistent with B10 — "the view/scope file is a real language"). This buys, for free: go-to-def on member packages/entities, find-refs, rename propagation, and validation that members actually exist. Crucially, it also **dissolves the nested-package entity-reference ambiguity** (§14.5 item 3): a `.ttrd` member like `prodeje.regional.faktura` is disambiguated by the real resolver (longest known-package prefix), not by a positional `split(".")` in YAML.

**Location & ownership.** `.ttrd` files live in the model repo (`model-ttr/`), versioned with the model, authored by the model/data architect. Agents (in `ai-models/agents/`, BA-owned) reference domains *by name* — preserving the existing ownership boundary.

**Concrete syntax (sketch; finalised in PD2):**

```
package domains            # the .ttrd file is itself in a package like any file

domain accounting {
    description: "Účetnictví a navazující doklady",
    tags: ["finance"],

    packages: [
        ucetnictvi,            # recursive: pulls ucetnictvi and ucetnictvi.*
        obchodni_doklady
    ],

    entities: [
        artikl.er.entity.artikl   # individual entity from an otherwise-excluded package
    ]
}
```

**Membership semantics.**
- `packages:` entries are **recursive** (B20): each pulls the named package and all descendants.
- `entities:` entries are individual objects (the "load one entity from a package you don't want wholesale" case the agent schema already supports), each a normal cross-reference resolved by the standard resolver.
- A domain's **resolved package set** = the recursive closure of `packages:` ∪ the packages owning each `entities:` entry. This closure is what the resolved-packages artifact (§14.6) records per domain.

**New diagnostics (finalised in PD3):**

| Code                              | Severity | Trigger                                                                 |
| --------------------------------- | -------- | ----------------------------------------------------------------------- |
| `ttr/domain-member-not-found`     | Warning  | A `packages:`/`entities:` member doesn't resolve                        |
| `ttr/domain-empty`                | Warning  | A domain has no members                                                 |
| `ttr/duplicate-domain`            | Error    | Two domains share a name across the project                             |
| `ttr/domain-redundant-member`     | Info     | An `entities:` entry is already covered by a recursive `packages:` member |
| `ttr/wrong-file-kind` (extended)  | Error    | `.ttrd` with no `domain` block, or a `domain` block in a `.ttr`/`.ttrg`  |

### 14.5 Impact on `ai-models` (B14, B21, B22)

Nesting + domains force coordinated changes in the sibling repo. These are specified for PD5 and tracked there; summarised here so the design record is complete.

1. **Agent schema (`agents/agent.schema.json`).**
   - `shem.packages[*]` pattern widens from `^[a-z0-9_]+$` to `^[a-z0-9_]+(\.[a-z0-9_]+)*$` (allow dotted nested packages).
   - Add an optional `shem.domains: string[]` field (domain names). At least one of `packages` / `domains` required.
   - `shem.entities[*]` — the two-segment `^[a-z0-9_]+\.[a-z0-9_]+$` pattern is **no longer parseable positionally** under nesting. Preferred resolution: push entity-granularity scoping into domains (`.ttrd` `entities:`), and keep `shem.entities` only as a raw escape hatch validated against the resolved-packages artifact (not by `split`).
2. **Runtime expansion (B21).** The Golem bootstrap resolves `domains` → concrete package/entity sets at load time (matching the existing 5-min metadata refresh model), rather than CI flattening them statically. `GOLEM_PACKAGES` / `GOLEM_ENTITIES` either carry domain names for the pod to expand, or are expanded in-process on refresh. Decided: **runtime**.
3. **CI validator (`tools/validate_agents.py`).** Replace rule 2 ("package = top-level dir") and the `split(".",1)` entity parsing with a lookup against the **resolved-packages artifact** (§14.6) that Modeler emits. The validator stops reimplementing model structure in Python; it reads a file. This realises the v2 TODO already noted in that script.

### 14.6 Resolved-packages artifact (B22)

Modeler emits a small, deterministic JSON artifact describing the resolved package/domain structure of a model project, for non-TS consumers (notably `ai-models` CI). Canonical shape lives in [contracts §13](v1-1-contracts.md#134-resolved-packages-artifact); summary:

- Emitted by a CLI subcommand (`modeler resolve-packages <project-root> [--out <file>]`). The CLI default `<project-root>/.modeler/resolved-packages.json` is a gitignored build artifact; the `ai-models` consumer instead **commits** the snapshot at `model-ttr/resolved-packages.json` (non-ignored) via `--out`, and guards its freshness with `resolve-packages --check` in a dedicated workflow (decision §14.7 Q1).
- Contains: schema/format version, the configured `root`, the full list of packages (canonical qname + whether nested + owning directory), every entity's qname keyed for existence checks, and every domain with its **resolved recursive package set** and entity set.
- Deterministic ordering so diffs are stable; regenerating with no model change yields a byte-identical file.

`ai-models` CI generates (or pulls a published) artifact and validates `agents/*.yaml` against it: every `shem.packages` entry must be a known package; every `shem.domains` entry a known domain; every `shem.entities` entry a known entity. Nesting and `root`-elision are handled by Modeler, not re-derived downstream.

The agent-registry gate reads it directly and stays **Node-free**; a separate, path-filtered `model-ttr` workflow runs `modeler resolve-packages --check` to block drift. No generate-in-the-agent-gate path. See plan PD4.6 / PD5.6.
2. **`.ttrd` placement.** One `domains.ttrd` at the model root vs one file per domain vs a `domains/` directory. Leaning one-file-per-domain (mirrors "one graph per `.ttrg`", B13). Settle in PD2.
3. **`shem.entities` future.** Keep it as a raw escape hatch, or deprecate it in favour of domain `entities:` once domains ship. Decide after PD5 lands and we see whether any agent still needs the raw form.
4. **`cnc` de-duplication (B23).** The doubled `cnc.cnc.role.*` is retained now; the option-(b) cleanup needs its own migration (rewrite live `model-ttr` references) and an ai-platform coordination beat. Track separately; do **not** bundle into this increment.
5. **`root` interaction with the agent registry.** If `[packages].root` is ever set for `model-ttr`, confirm the elision rule keeps existing bare agent `packages:` entries valid (it should, by §14.2). Add a regression fixture in PD5.

## 15. Addendum (2026-07-03) — family-wide `.ttrl` view-state sidecar (amends B2, §6.2 `layout`, §6.5)

> Recorded per decisions **C3-h** and **H-2c** of the TTR-P design effort ([`docs/ttr-p/design/00-control-room.md`](../../ttr-p/design/00-control-room.md), decision log 2026-07-03). This is a **design amendment, not shipped behaviour**: the v1.1 in-file `layout` block remains the implemented mechanism until the migration lands.

### 15.1 The decision

Layout — and presentation/view state generally — moves out of the program/graph file into a **per-document sidecar** with a **single family-wide grammar and extension**: `.ttrl`. The pairing is by filename: `x.ttrg` + `x.ttrl`, and on the TTR-P side `x.ttrp` + `x.ttrl`, `report.ttr.sql` + `report.ttrl`. One view-state scheme serves both languages (H-2c: "one layout scheme family-wide").

Consequences for this document:

- **B2 amended.** `.ttrl` was "removed in v1.1, replaced wholesale by `.ttrg`". The *extension* is now reclaimed with **new semantics**: no longer a project-wide layout aggregate, but a per-document view-state sidecar. This is **not** a revert of D4 — D4 killed the project-global `.modeler/layout.ttrl` aggregate, and per-document pairing preserves "layout travels with the document".
- **§6.2 / §6.5 amended.** The `layout` property inside `.ttrg` is superseded by the sidecar; TTR-M migrates off the in-file block. `modeler/getLayout` / `modeler/setLayout` re-target the sidecar; `setLayout` rewrites the sidecar **wholesale** (writer isolation — no `WorkspaceEdit` surgery inside graph text).
- **Scope widens.** The sidecar carries view state generally: node positions now; collapsed containers, annotations, edge routing later.

### 15.2 Why (from C3-h)

1. In-file coordinates churn every diff of the host document; for TTR-P, where the graphical surface is co-primary, this is untenable — and TTR-M inherits the same fix for consistency.
2. TTR-P **bare-fragment** programs (`report.ttr.sql`) force the sidecar: their source text is never rewritten by the toolchain, so derived-container layout has nowhere else to live.
3. Writer isolation: a wholesale sidecar rewrite is simpler and safer than surgical edits interleaved with user text.

### 15.3 Accepted costs and follow-ups

- **Pair integrity** becomes a real concern: orphaned/missing sidecar diagnostics and atomic rename handling are a shared (family-wide) toolset — tracked as a G work item on the TTR-P side; the same machinery serves `.ttrg`.
- **Content schema** of `.ttrl` is designed **once, for both languages**, in the TTR-P **C1 graphical session** (not here). Until then, nothing in this repo changes at implementation level.
- **Migration**: `.ttrg` files with an in-file `layout` block need a one-shot extraction to sidecars; plan alongside the C1 outcome. §6.5's "no project-wide layout file" statement survives unchanged.
