# 4.4 — Completion (table/column names in SQL)

**Repo:** modeler. **Effort:** ~1–1.5 days. **Package:** `@modeler/lsp` +
resolver/config. **Desktop only.** **Pre-flight:** 4.3 done. Uses the **reverse**
namespace map (3.3.4 `sqlNameFor`) to offer correctly-qualified names.

---

- [x] **4.4.1 — Spec (red).** Integration tests: completion after `FROM ` offers
      `db` tables (qualified per the dialect + namespace map); completion after
      `SELECT ` / `<alias>.` offers columns of the in-scope/aliased table; trigger
      char `.` after an alias offers that table's columns.

- [x] **4.4.2 — Context detection (best-effort).** From the cursor position in the
      masked SQL, determine the completion context: table position (after
      `FROM`/`JOIN`) vs column position (after `SELECT`/`WHERE`/`alias.`). Use the
      token stream + nearest `SqlRefModel` scope; tolerate partial/broken SQL
      (lexer-first — don't require a full parse). Keep heuristics simple and
      documented.

- [x] **4.4.3 — Table candidates.** Offer `db` tables visible to the project; label
      with the TTR namespace, insert the **dialect-correct qualified name** via
      `sqlNameFor(namespace)` + dialect quoting (§6.2) — e.g. tsql `dbo.Orders`,
      postgres `public.orders`. Include description in the completion detail.

- [x] **4.4.4 — Column candidates.** From the scope's in-view tables/aliases, offer
      their columns (type + description as detail). For `alias.` prefer that
      table's columns only. CTE/derived columns included when in scope.

- [x] **4.4.5 — Quoting/casing.** Insert identifiers with the dialect's
      quoting/casing rules (bracket vs double-quote; preserve TTR symbol casing or
      fold per dialect). Test one case per dialect.

- [x] **4.4.6 — Green; lint; commit.** Desktop-guard the provider.
      ```bash
      pnpm --filter @modeler/lsp --filter @modeler/integration-tests test
      pnpm --filter @modeler/lsp lint
      ```
      Commit: `Section embedded-sql: SQL completion`.
