# S0.2 — Real-query parse + span-fidelity evaluation

**Repo:** modeler (scratch `packages/sql-spike`). **Effort:** ~0.5–1 day.
**Gate:** a representative corpus of *actual* project T-SQL **and** Postgres
**lexes 100%** and **parses ≥ the agreed threshold** (proposed: ≥ 85% of
statements to a tree with no fatal error; tune with the owner); identifier tokens
carry tight `line/column/start/stop` spans; every parse failure is catalogued as
the lazy-patch backlog.

**Pre-flight:** S0.1 PASS (generated lexers/parsers available).

---

- [ ] **S0.2.1 — Assemble the corpus.** Collect real `sourceText` SQL from the
      project (and representative hand-written T-SQL/Postgres if `.ttr` coverage is
      thin) into `packages/sql-spike/corpus/{tsql,postgres}/*.sql`. Aim for
      breadth: plain SELECT, JOINs, CTEs, subqueries, `dbo.`/`public.` qualifiers,
      `@params`/`$1`, window functions, `[bracketed]`/`"quoted"` identifiers.
      Record the count per dialect.

- [ ] **S0.2.2 — Lex-coverage harness.** `src/lex-coverage.ts`: lex every corpus
      file; assert **zero** error tokens / no lexer exceptions. Any lexer failure
      is a gate blocker (lexer-first depends on 100% lex). Output a per-file table.

- [ ] **S0.2.3 — Parse-coverage harness.** `src/parse-coverage.ts`: parse every
      file with the **default error strategy** (NOT `BailErrorStrategy` — DESIGN
      §12.3); attach an error listener that counts syntax errors per file. Report
      the % of files that parse with zero errors and the % that produce a usable
      (partial) tree. Compare against the threshold.

- [ ] **S0.2.4 — Span-fidelity probe.** For a handful of queries, walk to a known
      column reference (e.g. `u.email` in `SELECT u.email FROM users u`) and assert
      the token's `line` / `column` / `start` / `stop` indices point exactly at
      `email`. This proves the §8 source map and go-to-def are feasible. Record 3–5
      concrete (token → expected span) samples in the report.

- [ ] **S0.2.5 — Catalogue failures → lazy-patch backlog.** For every file that
      failed to parse, record the construct that broke it (one line each) in
      `spike-report.md` § "S0.2 backlog". This is the prioritised list for DESIGN
      §12.7 lazy patching — it is a deliverable, not a defect log.

- [ ] **S0.2.6 — Gate verdict.** Append `spike-report.md` § "S0.2": lex %
      (must be 100%), parse % vs threshold, span samples (pass/fail), corpus
      sizes, backlog count. PASS only if lex is 100% and parse meets the threshold
      on **both** dialects. Borderline parse % → discuss threshold with owner
      before declaring PASS/FAIL (the feature tolerates imperfect parsing, but the
      threshold guards "useful semantics later").
