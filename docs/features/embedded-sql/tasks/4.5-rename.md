# 4.5 — Rename across the boundary (edit-mode dependent)

**Repo:** modeler. **Effort:** ~1 day. **Package:** `@modeler/lsp` +
`@modeler/edit`. **Desktop only.** **Status: CONDITIONAL** — renaming a `db`
symbol must rewrite its usages *inside* SQL text, which requires the
`WorkspaceEdit` synthesizer. Per DESIGN/architecture, edit synthesis is a **v1.1
edit-mode** capability (`@modeler/edit` is a placeholder until then). **Do not
start until edit mode lands**; this file is the spec for when it does.

---

- [ ] **4.5.1 — Pre-flight gate.** Confirm `@modeler/edit` can synthesize a
      `WorkspaceEdit` over a `.ttr` file (v1.1). If not yet available, **stop** —
      this stage is blocked. Record the blocker and leave Phase 4 at 4.4.

- [ ] **4.5.2 — Spec (red).** Rename a `db` column/table → all TTR references AND
      all in-SQL usages (across files) are rewritten; the SQL text inside the
      `"""…"""` block is edited in place with correct dialect quoting; surrounding
      SQL is untouched (trivia preserved).

- [ ] **4.5.3 — Build edits from the SQL reference index.** Reuse the 4.3 index to
      enumerate in-SQL usages of the symbol; for each, compute the file range of the
      identifier (source map) and a replacement honoring the block's dialect
      quoting/casing. Add to the `WorkspaceEdit` alongside the TTR-side edits.

- [ ] **4.5.4 — Qualified-name care.** Renaming a table must update only the table
      segment of a multi-part name (`dbo.Orders` → `dbo.Invoices`), not the
      schema/db qualifier. Renaming a column must not touch an identically-named
      column on a different table. Test both.

- [ ] **4.5.5 — Prepare-rename guard.** Implement `prepareRename` so rename is
      only offered on a resolved `db` ref (not on keywords, params, CTE names, or
      unresolved identifiers).

      pnpm --filter @modeler/lsp --filter @modeler/edit lint
