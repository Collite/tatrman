# Stage E ‚Äî `ai-models` repo migration + metadata-service check

Goal: migrate the **actual model content** in the `ai-models` repo (`~/Dev/ai-models`, folder
`model-ttr/`) to the Phase 0 conventions, regenerate its `resolved-packages.json`, and **verify
ai-platform's metadata service loads the migrated files**.

This is the real-content counterpart to Stages A‚ÄìC (which migrate the modeler's own
fixtures/tooling). It runs **last** because it needs the modeler 3.0 CLI (with `def area` + `.ttrm`)
and the ai-platform metadata loader branch from Stage D (D4) that understands `.ttrm` + `area`.

Scope, verified against `~/Dev/ai-models/model-ttr` (2026-06-24):
- **41 `.ttr` files** across package dirs (`artikl/`, `ucetnictvi/`, `prodeje/`, ‚Ä¶ each with
  `db.ttr` / `er.ttr` / `cnc.ttr` / `patterns.ttr`). ‚Üí rename to `.ttrm`.
- **2 `.ttrd` domain files**: `domains/accounting.ttrd`, `domains/obchod.ttrd` (each a single
  `domain <id> { description, tags, packages: [...] }` block). ‚Üí convert to `def area`.
- **No `schema map`** anywhere (model uses only `db`/`er`/`cnc`, per `model-ttr/modeler.toml`
  `[schemas] declared`). **The `binding` rename does not touch this repo.**
- `model-ttr/resolved-packages.json` is generated by the **Modeler CLI** (`modeler resolve-packages
  model-ttr --out ‚Ä¶`), driven by the repo `justfile`. ai-models' own Python tools
  (`tools/validate_agents.py`, `tools/gen_appset_agents/gen.py`) only scan `agents/**/*.yaml` and
  consume `resolved-packages.json` ‚Äî they do **not** read `.ttr` files, so they need no
  extension edits.
- CI: `.github/workflows/validate-model-ttr.yml` (resolved-packages drift gate), `validate-agents.yml`.
- `fabric-infra` reads this dir via `METADATA_GIT_SUBDIR=model-ttr`, `METADATA_GIT_KIND=ttr`.

Prereq: Stages A‚ÄìC merged; modeler 3.0 CLI buildable locally; ai-platform loader branch (D4)
available. Work on an `ai-models` feature branch.
