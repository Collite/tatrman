## 2. Decision review — confirmations

These I examined and **confirm**; reasoning holds and no better alternative was left unexplored:

- **A2 (separate TTR-family language)** — imperative dataflow vs declarative metadata is a real category difference; family ties preserve model resolution. Right call.
- **T2 complete package** — ports over edges, named+default, multicast, no implicit union, FS/SS/FF, error node-level both modes, control-hard-on-effect, events-as-model/v2-as-machinery, acyclic v1. Internally consistent; the events reframe (dependencies = event wirings) is a good foundation banked cheaply.
- **T9 movement model** — Charon moves-only, Load/Store/Transfer/Index distinct, Materialize as macro, engine-crossing = Store+Transfer+Load. The correction round (your points 1–13) removed a real misunderstanding and the resulting model is sharper than the pre-correction one.
- **Physicality spectrum + fixpoint, no logical/physical program split** — grounded in your own failed experiment with layered graphs; the "can this engine process it now?" formulation is the cleanest statement of the whole model.
- **Engine-relative primitive/macro + two rewrite kinds** — dissolves a false dichotomy; caveat in §3.3 below.
- **T3=γ** — proven by the strict-superset argument (PL ⊋ PlanNode); avoids wire-format coupling; two-tier Calcite reuse is exactly right. T5-a=adapt follows coherently.
- **NULL = canonical SQL 3VL** — forced by A4, correctly identified as a hard constraint, not a preference. Cost flag in §3.4.
- **Load/Store terminology** — the ETL "load=into warehouse" collision is real but the memory-boundary clarity wins; documented, eyes open.
- **G-b (WS-LSP + Designer server)** — the repo-attachment argument is decisive: once the Designer always needs the model repo, backendless bought nothing. The cascade (no KMP, no TS parser, no Python target, no conformance harness, one LSP over stdio+WS) is a large, genuine simplification. Confirm.
- **Table instance = SSA-for-storage** — load-bearing and correct; anonymous-on-edges / explicit-on-envelopes scoping is the right economy.

## 3. Decision review — pushback & revisits

### 3.1 A1-bis: all five surfaces ship in v1 — **revisit before C opens**
This is the decision I'd challenge hardest. The rationale (more surfaces = stronger pressure on a surface-agnostic core) justifies *designing* five surfaces, not *shipping* five. Note the internal tension: A4 defines "done" as the hero scenario in **≥2** surfaces, while A1-bis commits to building 5 — the success criteria don't require the commitment. The NL-like surface in particular is a controlled-grammar design + parser effort with weak pull from the v1 personas (engineer + analyst).

**Option not on record:** *design 5, ship 2–3* — sketch all five against the internal model during C0 (this is what pressure-tests the core), implement graphical + flow-DSL (the two full-coverage candidates, per Q3), and let SQL-like/pandas-like/NL follow as fast-follows once the core is proven. The isomorphism test survives; the v1 critical path shrinks by months. Recommend re-opening at C0.

### 3.2 v1 placement — an undecided fork being treated as decided
Containers bear the execution target (T9); no optimizer in v1. But **who creates containers and assigns targets in v1** was never explicitly decided. The design-space map leans "manual/explicit engine assignment" (E3) and Tatrman's optimizer-creates-containers is v2/Z — but nothing in the decision log says *"v1: the author draws containers and assigns engines by hand."* This is load-bearing for C (surfaces must be able to express containers + targets), for the Designer (container UX), and for T8 (does the fixpoint ever *move* nodes between authored containers in v1, or only via the T5-b escalation?). Small decision, big fan-out — make it explicit at the T6/T4 session.

### 3.3 T8 fixpoint: no termination/confluence story yet
The rewrite order (sugar → function-lowering → node re-placement) is settled, but the fixpoint now spans **two levels** (node + expression) plus a re-placement escalation that can *itself* create new engine-crossings (Store+Transfer+Load) that need normalizing. Nothing yet argues termination (does re-placement ever oscillate?) or confluence (do rule orders converge on the same graph?). Not urgent for concept work, but it should be a named T8 work item before implementation — e.g. a strictly-decreasing measure ("count of (node, container) capability misses") and a rule-stratification argument. T8-b is logged as open; I'd promote it from "open nuance" to "required before compiler build."

### 3.4 Whole-node re-placement can be needlessly coarse
T5-b: one unsupported function moves the **whole node** to another engine. For a `Project` computing 20 columns where 1 function is unsupported, that ships the entire dataset across an engine boundary for one column. **Option not discussed:** *node fission* — split `Project` into `Project(19 supported)` ∘ `Project(1 unsupported)` first, then re-place only the small one. This stays entirely within "node granularity, never mid-expression" (it's an ordinary node-level rewrite, like HAVING→Aggregate+Filter), so it doesn't reopen the closed fork — it's an extra rewrite rule that makes the escalation cheaper. Worth adding to the T8 rule catalogue.

### 3.5 FF "atomic co-finish" — implementability unvalidated
FF was adopted for atomic multi-output publish ("both tables or neither"). Across **two engines** (e.g. an MSSQL table + a file store), that's a distributed transaction — there is no cheap atomic commit. Realistically FF degrades to staging + best-effort swap + compensation, which is F territory and touches the T2-c retry/compensation deferral. FF stays in the model (correct), but F must own an honest answer to "what does FF actually guarantee cross-engine," and v1 might restrict FF to same-engine outputs. Flag it into F's brief now so it isn't discovered during implementation.

### 3.6 Canonical NULL: the pandas bill
Enforcing SQL 3VL on pandas is substantially harder than on Polars (NaN≠None, groupby silently drops NaN keys, NULL-key join semantics differ, NaN forces float upcasts). Polars has real null semantics and gets close for free. **Option not on record:** *v1 engine pair = SQL + Polars; pandas becomes a later target.* The brief lists both, but FI-2/A4 need only ≥2 engines, and the hero scenario names Polars. Dropping pandas from v1 removes the worst codegen burden with zero loss to the success criteria. Decide consciously in E rather than inheriting "pandas because the brief said so."

## 4. Options space not yet walked (undiscovered, not just undecided)

Beyond the tracked open items (T6, T4/Q7, T5/T7/T10 leftovers, G ratifications, TTR-designer convergence, Designer-server auth), these have **no options catalogued anywhere** yet:

1. **Security/RLS across engines.** Kantheon enforces RLS via Argos per query. A PL program can `Store` from an RLS-governed SQL source, `Transfer` via Charon, `Load` into Polars — at which point RLS is gone. Where does policy enforcement live for a cross-engine program: compile-time refusal, propagated row filters into every island, or "trusted pipeline" scope? Nothing in the docs touches this. Even if the v1 answer is "PL programs run as a trusted principal," that should be a recorded decision, not an omission. → new open question, lands in E/F.
2. **What "identical results" means (A4 verification).** Cross-engine equality is not trivial: float precision, collation (the Calcite-translator session shows exactly how gnarly this gets), NULL ordering in Sort, unordered results without Sort, timezone/datetime precision. A4 needs a defined equivalence procedure (canonical ordering + schema fingerprint + typed comparison with tolerances) — this is also the seed of the conformance test harness for E. Currently nowhere.
3. **F's option space generally.** F1 (new service vs extend Theseus vs Pythia's DAG executor) and F5 (minimal orchestrator: since container-collapse *produces* the orchestrator graph, v1 could get away with a small embedded runner that executes containers in FS order and calls Charon between — no platform service at all) are listed as one-liners in the map but never diverged. F is on the v1 critical path (FI-2) and is now the least-designed v1-mandatory piece.
4. **Lineage.** Tatrman's graph had lineage edges; PL's model has data/control/error ports and lineage was silently dropped. Probably correct — static column-level lineage is *derivable* from the graph + expressions and shouldn't be authored — but that's a decision worth one line in the log rather than an accident.
5. **Compiler diagnostics as a designed surface.** Split-with-warning, capability misses, placement previews — the warning/diagnostic vocabulary is becoming user-facing product surface (Designer server exposes it). Nobody has enumerated it. Cheap to seed as part of T6 (each manifest miss = a diagnostic).
6. **qname v4.0 alignment (cross-project).** The modeling language just went through the model/schema/binding rename (grammar 4.0). PL's D workstream and the `ttr-metadata` component must be specified in v4.0 vocabulary; `01-design-space-map.md` D1 still uses pre-4.0 phrasing. Also, T4's proposed rename of Tatrman's `TableContainer` to "*schema*/namespace" now collides with v4.0's narrowed meaning of *schema* (db-only SQL namespace) — "namespace" or "storage group" is the safer pick.

## 7. Overall assessment

The effort is in good shape: the decisions made are internally consistent, the rejections are recorded, and the two ideas that organize everything — physicality-spectrum-with-fixpoint and engine-relative primitivity — are both simple and genuinely original as a combination. The exposure is not in what was decided but in what hasn't been looked at: **F is v1-mandatory and undesigned**, the **verification semantics of A4** are undefined, and **one scope decision (five surfaces) is larger than its own success criteria require.** Address those three and the remaining path to C/D/E is mostly execution.
